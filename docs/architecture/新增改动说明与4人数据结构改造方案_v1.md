## 3. 目前“能否 4 人可用”的真实状态

### 3.1 已具备

- 数据库层已经具备多用户隔离所需主字段（`owner_user_id`）。
- 数据库层已经具备项目共享成员关系（`project_member`）。

### 3.2 还未完成（必须补）

以下是功能可用的硬门槛，目前尚未完成：

1. 认证与会话
- `login/logout/me` API
- token 签发与校验

2. 仓储层统一用户上下文
- 当前读写仓储尚未按 `owner_user_id` 过滤
- 当前插入尚未强制写入当前用户 `owner_user_id`

3. 权限校验
- 项目共享写入需校验 `project_member`
- 非成员访问共享项目应返回拒绝

4. 每用户参数
- 理想时薪目前仍走旧 `user_profile` 路径
- 需要迁移为以 `users.ideal_hourly_rate_cents` 为准

### 3.3 本次已完成的 P0 代码落地（新增）

1. 当前用户上下文
- 新增 `CurrentUserContext` 与 `SQLiteCurrentUserContext`。
- 默认读取当前激活用户；若未设置，自动回落到首个 active 用户。

2. 仓储层 owner 隔离
- 写入仓储：`createProject/createTimeLog/createIncome/createExpense/createLearning/createTag` 均写入 `owner_user_id`。
- 读取仓储：Overview、Recent、Tag、Metrics、Review 查询按 `owner_user_id` 过滤。
- 备份仓储：`backup_record/restore_record` 按 `owner_user_id` 写入与读取。

3. 项目共享基础权限
- 项目列表/详情/可选项目支持：
  - `project.owner_user_id = currentUserId`
  - 或 `project_member` 中包含当前用户
- 项目写入分配（时间/收入/学习）会校验项目是否可访问，防止越权 project_id 注入。

4. 理想时薪改为用户级
- Metrics 读取/写入理想时薪改为 `users.ideal_hourly_rate_cents`。

5. 启动期兜底
- `LifeOsOpenHelper` 增加默认用户补齐和 owner 回填兜底，避免新装/升级后无 current user。

## 4. “怎么改更好”建议（按 4 人目标）

### P0（必须，先做）

1. 引入 `CurrentUserContext`（App 内统一用户上下文）
2. 所有写入接口强制写 `owner_user_id = currentUserId`
3. 所有读取接口默认加 `WHERE owner_user_id = currentUserId`
4. 项目读取改为：
   - 自有项目：`project.owner_user_id = currentUserId`
   - 共享项目：`EXISTS project_member(project_id, currentUserId)`

### P1（必须，1 周内）

1. 落地 auth API：
   - `POST /auth/login`
   - `POST /auth/logout`
   - `GET /auth/me`
2. `users.password_hash` 改为真实哈希（bcrypt/argon2），禁止明文。
3. 将理想时薪读写迁移到 `users` 表。

### P2（建议）

1. 解决全局唯一键问题（重建表）：
   - 当前 `project.name`、`tag.name` 是全局唯一，不适合多人。
   - 建议改为：
     - `UNIQUE(owner_user_id, name)` for project
     - `UNIQUE(owner_user_id, name, scope)` for tag
2. 审计表补 `actor_user_id`（谁执行）与 `owner_user_id`（谁的数据）。

---

## 5. 风险提示（当前结构）

1. `V003` 只完成了“结构基础”，不是完整多人功能。
2. 若直接开放 4 人使用而不做仓储层过滤，会出现串数据风险。
3. `users.password_hash` 在迁移中回填值为占位符 `__SET_ME__`，上线前必须替换为真实哈希策略。

---

## 6. 建议的下一执行顺序（可直接开始）

1. 先做仓储层 `owner_user_id` 强制写入与查询过滤（最高优先级）。
2. 再做最小 auth（登录态与 `currentUserId` 注入）。
3. 再接项目共享权限。
4. 最后做唯一键重建迁移（project/tag）。
