# LifeOS 基础架构与数据设计 v1

## 1. 目标

本设计只解决“长期可用的底层”，不进入具体业务交互开发。

核心目标：
- 支撑 10-30 年连续数据积累
- 支撑高频输入（每日记录）与低频分析（月/年趋势）
- 数据结构稳定，可持续迁移升级
- 单用户优先，未来可平滑扩展多用户

## 2. 架构总览（单用户 Android 本地优先）

采用分层架构：

1. Presentation（UI/交互）
- 录入页、日结页、看板页、设置页
- 不直接访问数据库

2. Application（用例层）
- 解析输入、写入记录、计算指标、生成周报
- 事务边界与业务规则在此层集中

3. Domain（领域层）
- 实体模型（时间、项目、收入、支出、学习、标签）
- 规则模型（时间负债、自由度、ROI、学习转化）

4. Data（数据层）
- SQLite（主存储）
- DAO/Repository（读写抽象）
- Local Backup（JSON/CSV + SQL dump）

5. Input Adapter（可插拔）
- 当前以手动录入/模板录入为主
- 预留 AI 接口，但不实现复杂 AI 流程
- 任何外部解析结果都必须走应用层校验

## 3. 数据域边界

核心域：
- 时间记录（time_log）
- 项目（project）
- 收入（income）
- 支出（expense）
- 学习（learning_record）
- 标签（tag）
- 日结（daily_review）

辅助域：
- 绑定关系（多对多 join 表）
- 指标快照（metric_snapshot）
- 预留外部输入接口（source / confidence 字段）
- 备份与恢复记录（backup_record / restore_record）
- 变更审计（audit_log）

## 4. 关键设计决策

1. 未绑定项目进入“公共池”
- 时间/收入/学习允许不绑定项目
- 通过 `is_public_pool` 字段显式标注，避免丢失价值

2. 跨天时间记录原生支持
- 存 start_at / end_at（UTC 时间戳或 ISO8601）
- 不靠“同一天假设”，直接按真实时间计算时长

3. 标签双层机制
- 系统标签（不可删除，可停用）
- 用户标签（可新增、可停用）

4. 可演化优先
- 所有主表使用 `created_at/updated_at`
- 软删除字段 `is_deleted`
- 独立 schema 版本管理（migration 表）

5. 读写分离思路（逻辑层）
- 原始事实表：只存事实记录
- 快照表：存按日/周/月聚合指标，提升大盘查询性能

## 5. 数据规模预估（单用户 30 年）

按保守估计：
- time_log：每天 8 条 * 365 * 30 ≈ 87,600
- income/expense：每天 3 条 * 365 * 30 ≈ 32,850
- learning_record：每天 1 条 * 365 * 30 ≈ 10,950
- join 表 + audit + snapshot：总计约 30-80 万行

SQLite 在该量级可稳定运行，关键是索引与归档策略。

## 6. 索引与性能策略

必备索引：
- 按日期范围查询：`date`, `start_at`, `end_at`
- 按项目聚合：`project_id` + `date`
- 按标签筛选：join 表中的 `(tag_id, target_type, target_id)`
- 软删除过滤：`is_deleted`

策略：
- 月度指标写入快照，避免每次全量回算
- 历史年数据可按“归档导出 + 留摘要”降低 IO

## 7. 备份与恢复策略（本地优先）

备份层级：
- Daily 增量（当天变化）
- Weekly 全量（数据库文件）
- Monthly 归档（数据库 + JSON 导出）

恢复原则：
- 允许选择任意备份点恢复
- 恢复动作写入 restore_record
- 恢复后强制重建指标快照

## 8. 迁移策略

采用版本化 SQL 迁移：
- `migration(version, name, applied_at, checksum)`
- 升级时顺序执行 `Vxxx__*.sql`
- 每次迁移必须可重放或幂等检查

破坏性变更原则：
- 先新增新列/新表并双写
- 数据回填完成后再切换读路径
- 最后清理旧字段

## 9. 观测与审计

记录范围：
- 核心业务数据的增删改（含来源：手动/外部解析/导入）
- 关键指标计算任务状态
- 备份与恢复结果

目标：
- 出错可追溯
- 外部解析误差可复盘

## 10. 分阶段落地建议

Phase 0（当前）：
- 固化 schema + migration + backup 结构

Phase 1：
- 时间/项目/收支/学习 CRUD
- 核心指标离线计算

Phase 2：
- 外部解析接入（规则引擎或 AI 都可）
- 简单确认流（先不做复杂任务队列）

Phase 3：
- 高级报告与趋势预测
- 策略建议（不是自动执行）
